## Context
Today every key in the kms-wrapper is a single secp256k1 keypair generated by `internal/plugin/path_keys.go:createKey`. Both `/sign/evm` and `/sign/cosmos` accept any `key_path` and write a pre-hashed 32-byte `input` to the same plugin sign endpoint (`kms/sign/<path>`), with no `hash_algorithm` parameter. The path no longer encodes chain (after `update-key-path-scheme`), and there is no per-key metadata expressing which chain(s) the key is allowed to sign for.

We want an explicit, persisted, plugin-enforced capability tag — not a documentation-only convention — so:
- Operators can create a Cosmos-only key for a system that only ever signs Cosmos and gain a real guard against accidental EVM signing.
- Operators can create a dual-capable key when an identity legitimately spans both ecosystems.
- Audit logs cleanly separate "principal denied" 403s from "key not authorized for this chain" 403s without operators having to grep payload shapes.

## Goals / Non-Goals
- Goals
  - Required `chains` field at create — no silent default, every key carries explicit intent.
  - Plugin-side enforcement at the trust boundary, mirroring the existing pattern for key-path format enforcement.
  - Gateway-side enforcement as the fast-fail path so misrouted requests never reach Vault.
  - Expand-only mutability: a key can grow capability (`[evm]` → `[evm, cosmos]`) but never lose it.
  - Response shape mirrors the tag — only the enabled chains' derived addresses appear.
- Non-Goals
  - Per-call chain selection. Once a key is tagged, the tag governs every call; there is no per-request override.
  - Auth roles or per-token chain capabilities. The tag lives on the **key**, not on the **principal**.
  - Migrating existing keys. The migration path is "recreate at the same path with explicit `chains`" — same as the path-rename migration that lands first.

## Decisions

### D1 — Required field, not a default
`chains` is required at create. Missing → HTTP 400.
- **Why**: defaults rot. A `["evm", "cosmos"]` default would survive months of copy-pasted scripts and then silently produce dual-capable keys when an operator typed `POST /keys '{"path":"..."}'` expecting the same shape as before. Required-and-explicit catches that mistake at the call site.
- **Alternatives considered**:
  - Default to `["evm","cosmos"]` (backwards-friendly). Rejected — silent permissive defaults are exactly what this change exists to remove.
  - Default to `[]` (deliberately useless). Rejected — degenerate keys with no capability create worse-than-useless audit noise.

### D2 — HTTP 403 (not 400 or 422) on chain mismatch at sign time
Rejected requests at sign time return 403 with a key-and-chain-specific body. The error body **must** name the key, the attempted chain, and the allowed chains. Structured `slog` warn carries the same triple.
- **Why**: the codebase already uses 403 for every policy refusal (bearer-token reject, Vault `permission denied` → `types.ErrPermission`). The request body is well-formed and the caller is authenticated — what's being refused is *resource policy*, which is exactly what 403 means in this stack. Using 400 would conflate "input shape is wrong" with "policy says no".
- **Alternatives considered**:
  - 422 Unprocessable Content. Rejected — semantically tighter ("syntactically valid, semantically refused"), but rare in this codebase. Introducing a third class of denial would force every audit dashboard to learn a new code without offsetting benefit.
  - Warn-only enforcement (sign anyway, just log). Rejected — defeats the entire purpose of the change.

### D3 — Plugin and gateway both enforce
Same pattern as the existing key-path-format enforcement (key-path-policy spec: "Plugin enforces key-path format independently of the gateway"). The plugin is the trust boundary — direct Vault clients (operators, alternative CLIs, future automation) must hit the same wall. The gateway check is the fast-fail path so the network round-trip is avoided for misrouted requests.
- Both layers return 403 with the same body shape, mediated by the plugin's `logical.ErrPermission` → 403 mapping that `internal/vault/client.go` already performs.

### D4 — Tag is persisted on `KeyEntry`, canonicalized
`KeyEntry.Chains []string` stored as a sorted, lowercase, deduped slice. Validation accepts only members of the closed set `{"evm", "cosmos"}` — unknown values rejected at create with HTTP 400.
- **Why canonical**: stable serialization across plugin upgrades; `==` and `slices.Contains` work without normalization passes at sign time.
- **Why a closed set**: every additional chain expands the enforcement surface (signer code, address derivation, swagger enum). Adding new chains is a future spec change, not a free-form opt-in field.

### D5 — Expand-only PATCH (`add_chains`), no remove
`PATCH /keys/{path}` accepts `{ "add_chains": [...] }`. Any request that would shrink the tag returns 400.
- **Why**: the audit story stays monotonic. "This key has been able to sign Cosmos since timestamp T" is a fact a security review can assert; "this key could sign Cosmos for six months then suddenly couldn't" creates a class of incidents (orphaned outstanding txs, signed-but-now-orphaned commitments) that the operator probably did not mean to take on.
- **Alternatives considered**:
  - Immutable (no PATCH). Rejected — the expand-to-dual case (originally `[evm]`, later "we need this same identity to sign Cosmos too") is realistic and forcing a new key + caller migration is a big tax for a small change.
  - Fully mutable. Rejected — non-monotonic policy on a signing key is the kind of thing that bites in postmortems.

### D6 — Response shape mirrors the tag
`evm_address` and `cosmos_address` appear in `POST /keys` / `GET /keys/info` / `GET /keys` responses **only** when the corresponding chain is in `chains`. `public_key_hex` is always present (it is curve-level, not chain-level).
- **Why**: surfacing a `cosmos_address` for an `[evm]`-tagged key invites callers to use it ("the API gave it to me; surely I can sign with it"), undermining the tag.
- **Trade-off**: response shape becomes conditional. Mitigated by:
  - The `chains` field is always present, so clients can branch on it.
  - The omitted-vs-empty distinction is consistent — addresses are simply absent when not applicable, never null/empty-string.

### D7 — CLI flag shape
- `kms-wrapper keys create --path <p> --chains evm,cosmos` (comma-separated repeatable). One flag, one value, easy to script.
- `kms-wrapper keys update-chains --path <p> --add cosmos` (single `--add` flag, comma-separated). Mirrors the PATCH body shape.
- Rejected: `--evm --cosmos` boolean pair (cute, but does not extend if a third chain is ever added).

## Risks / Trade-offs
- **Risk**: a follow-up that adds a third chain (Solana, Ed25519-based Cosmos chains, etc.) has to update the closed enum AND all the enforcement points. Mitigation: the closed set lives in one place (`pkg/types`), and the enforcement is centralized at the gateway and plugin sign paths. Adding a chain is a deliberate, reviewable change.
- **Risk**: existing tests across `internal/gateway`, `internal/plugin`, `internal/vault`, `internal/keyinfo` all create keys without a `chains` field; making the field required means every test fixture must update. Mitigation: tasks.md sequences the test fixture update before the request-shape change.
- **Trade-off**: response shape becomes conditional on `chains`. Callers that hard-code `response.cosmos_address` will fail on `[evm]`-tagged keys. This is by design — the alternative (always return both) defeats the purpose.

## Migration Plan
1. Land `update-key-path-scheme` first. This proposal assumes the new `{project}/{environment}/{username}` shape and updates examples accordingly.
2. Land this change. Operators and any consuming services must update create calls to include `chains`.
3. There is no in-place migration for existing keys. The project is pre-production; local dev rebuilds via `make scrub-env && make dev-down && make dev-up`. Any keys an operator wants to preserve are recreated at the same path with explicit `chains`.
4. Rollback plan: revert the PR. `KeyEntry` is stored as JSON, so an older binary will silently ignore the unknown `chains` field on read — the field is NOT auto-deleted on restart. It only disappears for an entry that the older binary rewrites (e.g. via `update-chains` or any future write path that re-marshals the struct); entries that are merely read keep their `chains` JSON bytes intact in storage. The secp256k1 key material is preserved either way.

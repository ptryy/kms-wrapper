## Why

The CLI already supports `kms-wrapper keys create|show|list` against Vault Transit, but the REST gateway exposes only `/health` and `/sign/*`. Operators, integration tests, and other services that talk to the gateway over HTTP today have no way to provision a key, fetch its derived EVM/Cosmos addresses, or enumerate keys under a prefix without shelling into a host and running the CLI. Adding three HTTP endpoints closes that gap and lets consumers stay on a single transport (and a single bearer-token surface) for the full key lifecycle.

## What Changes

- Add three authenticated HTTP endpoints to the REST gateway, all sharing today's bearer-token middleware and the existing `/sign/*` rate limiter:
  - `POST /keys` â€” create a secp256k1 Transit key at `{path}`. Idempotent â€” returns `200 OK` with `KeyInfo` plus `already_existed: true|false`. Mirrors `kms-wrapper keys create --path <p>`.
  - `GET /keys/info?path=<path>` â€” return the public key (hex), derived Ethereum address (EIP-55), and Cosmos bech32 address (default HRP `cosmos`). `404` when the key is not present. Mirrors `kms-wrapper keys show --path <p>`.
  - `GET /keys?prefix=<prefix>` â€” list bare key names under `{prefix}` (empty prefix â†’ top level). Response: `{"keys": ["..."], "count": N}`. Mirrors `kms-wrapper keys list --prefix <p>`.
- Reuse existing primitives from `internal/vault/client.go` (`CreateKey`, `GetPublicKey`, `ListKeys`) and the EVM/Cosmos address derivation helpers used by `printKeyInfo` in `cmd/kms-wrapper/root.go`. No new vault-plugin surface.
- Extend `pkg/types/types.go` with explicit request/response structs (`KeyCreateRequest`, `KeyCreateResponse`, `KeyListResponse`) and add swaggo annotations on the new handlers so the OpenAPI 3.0 spec generated by `make swagger` documents them. Update `docs/swagger.json`, `docs/swagger.yaml`, and `docs/docs.go` accordingly.
- Wire the new routes into `gateway.routes` behind `requestLogger â†’ rateLimit â†’ auth`, identical to the signing endpoints. No new config knobs, no new env vars.
- Error responses follow the existing `apptypes.ErrorResponse` shape (`{"error": "<message>"}`); validation errors return `400`, missing keys `404`, vault failures `500`. Never leak vault tokens or key material.
- **No breaking changes**: existing endpoints, request/response shapes, CLI flags, and config keys are unchanged.

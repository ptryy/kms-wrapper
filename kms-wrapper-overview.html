kms-wrapper

A self-hostable, multi-chain Key Management System gateway backed by HashiCorp Vault Transit Engine. Centralises transaction signing for EVM and Cosmos SDK chains — no private key ever leaves Vault.

Go 1.22+ HashiCorp Vault 1.10+ EVM (Ethereum / secp256k1) Cosmos SDK REST Gateway CLI Multi-tenant Docker Compose

Project Goal
🎯

Eliminate Plaintext Key Exposure

Replace private keys embedded in validator configs, relayer YAML, and app secrets with a single signing gateway. Keys live exclusively inside Vault's Transit Engine — they are never serialised, logged, or returned over any API.

🔗

Unified Multi-Chain Signing Layer

One binary, one REST API, one CLI handles both EVM (raw tx, personal_sign, EIP-712) and Cosmos SDK (SIGN_MODE_DIRECT, AMINO_JSON) signing. No more per-chain Vault integrations.

🏗

Self-Hostable, No Cloud KMS Dependency

Runs on a standard Linux VM behind an internal VPN. No HSM hardware, no AWS KMS, no GCP Cloud KMS. The only dependency is a reachable Vault instance (dev mode via Docker Compose for local use).

🏢

Multi-Tenant from Day 1

Key paths encode project/chain/username. Vault policies can be scoped per project, enabling different teams to share the same Vault mount without cross-tenant key access.

Key Path Convention
transit/keys/{project} / {chain} / {username}
project
Tenant/project scope. Vault policies are scoped to transit/*/project-a/* for isolation.
chain
Reserved values: evm, eth, mantra, cosmos, osmosis. Unknown chains pass through with a warning.
username
Identity of the signing entity: validator, relayer, alice, etc. Segments: [a-z0-9_-] only.
REST Gateway API
Method Endpoint Auth Description
GET /health None Vault connectivity status. Returns {"status":"ok","vault":"reachable"} or 503 degraded.
POST /sign/evm Bearer Token Sign EVM transaction. Accepts raw_tx (RLP hex), personal_message, or eip712_digest. Returns signed tx + r/s/v.
POST /sign/cosmos Bearer Token Sign Cosmos transaction. Accepts sign_doc (base64 protobuf or amino JSON). Returns base64 signature + compressed pubkey.
POST /keys/create Bearer Token Create a new ecdsa-p256k1 Transit key at the specified path. Idempotent.
GET /keys/show Bearer Token Retrieve public key, EVM address (EIP-55), and Cosmos bech32 address for a key path.
Project Use Cases
1

Cosmos Validator & IBC Relayer Signing

Validators and IBC relayers need to sign thousands of transactions per day. Instead of storing the operator key in a hot wallet file on the server, the relayer calls POST /sign/cosmos with the sign doc. The key stays in Vault; the server only holds a short-lived bearer token.

MANTRA Chain Cosmos Hub Osmosis SIGN_MODE_DIRECT AMINO_JSON
2

EVM Node Operator & MEV Bot Signing

EVM validators, MEV searchers, and automated trading bots require high-frequency raw transaction signing. POST /sign/evm accepts RLP-encoded unsigned transactions and returns fully signed EIP-155 transactions ready for broadcast — without the bot ever holding the private key.

Ethereum EVM-compatible L2s EIP-155 eth_sendRawTransaction
3

dApp Backend Server-Side Signing

Backend services that need to sign EIP-712 typed data (permit2, meta-transactions, gasless flows) or personal_sign messages for authentication can delegate signing to kms-wrapper without embedding keys in environment variables or secrets managers.

EIP-712 personal_sign Meta-transactions Permit2
4

Multi-Tenant DevOps Key Management

Platform teams managing multiple projects and chains use the CLI (kms-wrapper keys create/show) and Vault policy scoping to provision isolated signing keys per project. Each project gets its own key namespace (project-a/mantra/relayer) with no cross-tenant access.

Multi-tenant Vault Policy Isolation DevOps / InfraSec
5

CI/CD Pipeline Signing

Automated deployment pipelines that need to submit on-chain governance proposals, contract deployments, or upgrade transactions can call the gateway from CI (GitHub Actions, etc.) using a short-lived KMS_GATEWAY_TOKEN without storing long-lived private keys in CI secrets.

Project Benefits

🔐 Zero Key Exposure

HashiCorp Vault Transit Engine performs all signing server-side. Private keys are never serialised to disk, transmitted over the network, or returned in any API response. The security boundary is enforced at the cryptographic level.

⛓ Multi-Chain, Single API

One gateway handles EVM (raw tx, personal_sign, EIP-712) and Cosmos SDK (SIGN_MODE_DIRECT, AMINO_JSON) signing. Downstream services don't need chain-specific Vault integrations — one HTTP call, two ecosystems.

🏗 Self-Hostable, No Vendor Lock-in

Runs on any Linux VM on an internal VPN. No cloud KMS, no HSM hardware, no SaaS dependency. The only requirement is Vault 1.10+, which can itself run in dev mode locally or in a hardened production cluster.

🏢 Multi-Tenant by Design

Key paths encode project, chain, and identity. Vault policies scope per transit/*/project-a/*. Different teams share one Vault mount without risk of cross-tenant key access — critical for infrastructure shared across validator sets.

⚡ Single Binary Distribution

One compiled Go binary serves both as a CLI tool (key management, ad-hoc signing, health checks) and as the HTTP gateway server (kms-wrapper serve). Operators scp one file — no dependency installation, no container required.

🔌 Extensible Auth Interface

The AuthProvider interface is defined from day 1 with a stub AppRoleAuthProvider. Upgrading from static bearer token to Vault AppRole or mTLS requires only a new implementation — no gateway refactor needed.

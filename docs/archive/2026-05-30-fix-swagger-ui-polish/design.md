## Context

Verification of `fix-swagger-runtime-server-url` and `add-swagger-docs` produced three findings that aren't bugs but create friction:

1. `docs/swagger.json` ships with `"servers":[{"url":"http://localhost:8080/"}]`. The postprocessor correctly adds the `http://` scheme; the placeholder origin is intentional. But nothing in the repo tells a developer importing this file into Postman or an SDK generator why it says `localhost:8080` and what to change it to.

2. `requestOrigin()` accepts `X-Forwarded-Proto: https` or `http` and silently falls back to `http` for any other value (e.g. `ftp`, typos, malformed chains). When a reverse proxy is misconfigured to forward a wrong value, there is no observable signal in the gateway logs.

3. `GET /swagger/` triggers an HTTP 301 from inside the `http-swagger` library before the handler serves `index.html`. The redirect works in standard browsers, but 301s are permanently cached — if the path ever changes, clients that have cached this redirect are stuck. In environments where a strict reverse proxy or CSP blocks the `Location` header, the UI becomes unreachable.

All three changes are isolated to `internal/gateway/gateway.go` and `README.md`; no new dependencies, no config changes, no breaking API changes.

## Goals / Non-Goals

**Goals:**
- `GET /swagger/` returns HTTP 200 with the UI body — no redirect, no permanent cache entry
- Operators with debug-level logging enabled can see when an invalid `X-Forwarded-Proto` value was ignored
- `README.md` explains clearly that the static file's server URL is a placeholder and the runtime endpoint is the source of truth

**Non-Goals:**
- Replacing or vendoring `http-swagger` (the library remains; we intercept one path before it can redirect)
- Logging at warn/error level for unknown `X-Forwarded-Proto` — this is not an error condition, and production logs should stay clean
- Modifying the generated `docs/swagger.json` content (it is a generated artifact; the README is the right place for human-readable guidance)

## Decisions

### D1 — Rewrite path inside the `/swagger/` handler, don't add a redirect

**Decision**: In the mux handler closure for `GET /swagger/`, detect when `r.URL.Path` is exactly `/swagger/` and rewrite it to `/swagger/index.html` on the request before passing to `httpSwagger.Handler`. The handler then serves the index directly and returns 200.

```go
mux.Handle("GET /swagger/", http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
    if r.URL.Path == "/swagger/" {
        r2 := r.Clone(r.Context())
        r2.URL.Path = "/swagger/index.html"
        swaggerUI.ServeHTTP(w, r2)
        return
    }
    swaggerUI.ServeHTTP(w, r)
}))
```

**Rationale**: A server-side path rewrite is invisible to the client — it sees 200 with the UI body, no `Location` header, nothing to cache or block. Alternatives:

- *Separate `GET /swagger/index.html` registration*: would require duplicating the auth/rate-limit wrapping and risks the two handlers drifting.
- *`http.StatusSeeOther` (303) redirect*: still a redirect, still potentially blocked, not cacheable — better than 301 but doesn't eliminate the round-trip.
- *`http.StatusTemporaryRedirect` (307)*: same round-trip problem; avoids permanent caching but changes method semantics for POST (irrelevant for a GET-only route).

### D2 — `slog.DebugContext` for rejected `X-Forwarded-Proto` values

**Decision**: After the `switch` in `requestOrigin()` rejects an unknown value, add one `slog.DebugContext(r.Context(), "ignoring unrecognised X-Forwarded-Proto", "value", forwarded)`. The existing `scheme` variable already defaults to `http` before this point; no logic change, only observability.

**Rationale**: Debug-level is the right severity — this is not an error, and it fires on every request when a proxy is misconfigured, which would flood warn/info logs in production. Operators who need to diagnose proxy issues can enable debug level temporarily.

### D3 — README note only, no change to generated files

**Decision**: Add a "Static spec file" note in `README.md` under the API docs section explaining that `docs/swagger.json` contains a `localhost:8080` placeholder and pointing to the runtime endpoint for live use.

**Rationale**: `docs/swagger.json` is generated by `swag init` and post-processed; hand-editing it would be overwritten on the next `make swagger`. A YAML comment in `docs/swagger.yaml` would be stripped by re-generation for the same reason. `README.md` is the stable, human-maintained surface.

## Risks / Trade-offs

- **Path rewrite on `r.Clone`** — cloning the request is cheap but allocates. `/swagger/` is a low-traffic developer route, so this is not a performance concern.
- **Library internals** — the rewrite works because `httpSwagger.Handler` serves `index.html` when the path suffix is `index.html`. If a future version of the library changes this, the handler would need to be updated. The test added in tasks locks this behavior.

## Migration Plan

1. Update `gateway.go`: add path-rewrite closure and `slog.DebugContext` call
2. Update `README.md`: add static spec note
3. Add a gateway test asserting `GET /swagger/` → HTTP 200 (no redirect)
4. No deploy steps — purely in-process, no config, no migration
